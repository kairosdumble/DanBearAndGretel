주변 동승자 거리 기반 정렬 작업 진행 순서

전제

현재 프로젝트는 Tmap API를 이미 사용하고 있습니다.
Tmap API는 사용자가 검색한 장소의 이름, 주소, 위도, 경도를 가져오는 역할입니다.

이번 작업에서 새로 필요한 것은 Tmap API를 바꾸는 것이 아니라,
Tmap으로 얻은 좌표를 예약 DB에 저장하고,
동승자 목록 조회 시 사용자의 출발지 좌표와 DB 예약 출발지 좌표 사이의 거리를 계산해서 가까운 순서로 정렬하는 것입니다.


1. 현재 역할 정리

Tmap API 역할:

- 장소 검색
- 장소 이름 조회
- 도로명 주소 조회
- 위도/경도 조회

우리 백엔드 역할:

- 예약 생성
- 예약 목록 조회
- DB에 저장된 예약 출발지 좌표와 사용자가 선택한 출발지 좌표 사이의 거리 계산
- 가까운 순서로 예약 목록 정렬


2. DB 스키마 확장

현재 reservations 테이블에는 주소 문자열만 저장되어 있습니다.

- departure_location
- destination_location

거리 계산을 하려면 예약 출발지의 위도/경도 좌표가 필요합니다.
따라서 reservations 테이블에 아래 컬럼을 추가합니다.

- departure_lat DOUBLE PRECISION
- departure_lng DOUBLE PRECISION
- destination_lat DOUBLE PRECISION
- destination_lng DOUBLE PRECISION

수정 대상 파일:

- backend/db/schema.sql

이미 운영 중이거나 기존 DB가 있다면 CREATE TABLE만 수정해서는 기존 테이블에 컬럼이 추가되지 않을 수 있습니다.
따라서 ALTER TABLE도 같이 넣는 것이 안전합니다.

예:

ALTER TABLE reservations
  ADD COLUMN IF NOT EXISTS departure_lat DOUBLE PRECISION,
  ADD COLUMN IF NOT EXISTS departure_lng DOUBLE PRECISION,
  ADD COLUMN IF NOT EXISTS destination_lat DOUBLE PRECISION,
  ADD COLUMN IF NOT EXISTS destination_lng DOUBLE PRECISION;


3. 예약 생성 API 수정

예약 생성 시 장소 이름뿐 아니라 좌표도 함께 저장해야 합니다.

수정 대상 파일:

- backend/src/controllers/reservationController.js
- backend/src/services/reservation.service.js

현재 저장하는 값:

- departure_location
- destination_location
- departure_time

추가로 저장해야 하는 값:

- departure_lat
- departure_lng
- destination_lat
- destination_lng

백엔드에서는 좌표 값이 숫자인지 검증하는 것이 좋습니다.


4. 예약 생성 화면에서 좌표까지 전송

예약 생성 화면 파일:

- frontend/lib/features/nearbyMateDetail/screens/NearbyMateDetail.dart

프론트엔드의 Place 모델은 이미 Tmap 검색 결과에서 좌표를 가지고 있습니다.

Place 모델 파일:

- frontend/lib/features/home/screens/place.dart

Place 주요 필드:

- name
- roadAddress
- latitude
- longitude

예약 생성 요청 body에 아래 값을 추가합니다.

- departure_lat: _departure!.latitude
- departure_lng: _departure!.longitude
- destination_lat: _destination!.latitude
- destination_lng: _destination!.longitude


5. 홈 화면에서 선택한 출발지/목적지를 목록 화면으로 전달

홈 화면 파일:

- frontend/lib/features/home/screens/home.dart

현재는 홈 화면에서 출발지/목적지를 선택해도 NearbyMateList 화면으로 전달하지 않습니다.

현재 구조:

NearbyMateList()

수정 방향:

NearbyMateList(
  departure: _departure!,
  destination: _destination!,
)

NearbyMateList 생성자에서 Place departure, Place destination을 받도록 수정합니다.


6. 기존 예약 목록 API 확장

새 API를 만들지 않고 기존 API를 확장하는 방식이 덜 헷갈립니다.

현재 목록 조회 API:

GET /api/reservations/all

확장 후:

GET /api/reservations/all
-> 기존처럼 전체 예약 목록 조회

GET /api/reservations/all?lat=37.123&lng=127.123
-> 사용자가 선택한 출발지 좌표 기준으로 가까운 예약부터 정렬

즉, lat/lng 쿼리 파라미터가 있으면 거리 계산 정렬을 하고,
lat/lng가 없으면 기존 동작을 유지합니다.

수정 대상 파일:

- backend/src/controllers/reservationController.js
- backend/src/services/reservation.service.js


7. 거리 계산 방식

거리 계산 기준:

- 사용자가 홈 화면에서 선택한 출발지 좌표
- DB에 저장된 각 예약의 departure_lat / departure_lng

처음 구현은 Haversine 공식으로 충분합니다.
PostGIS 같은 지리 확장 기능은 지금 단계에서는 굳이 필요하지 않습니다.

백엔드 응답에는 각 예약마다 아래 값을 추가해서 내려주는 것이 좋습니다.

- distance_meters

정렬 기준:

- distance_meters ASC

즉, 가까운 예약이 먼저 오도록 정렬합니다.


8. 동승자 목록 화면 수정

목록 화면 파일:

- frontend/lib/features/nearbyMateList/screens/nearbyMateList.dart

현재는 아래 API를 호출합니다.

GET /api/reservations/all

수정 후에는 홈 화면에서 전달받은 출발지 좌표를 쿼리 파라미터로 붙입니다.

예:

GET /api/reservations/all?lat=${departure.latitude}&lng=${departure.longitude}

응답에 distance_meters가 있으면 목록에 함께 표시합니다.

표시 예:

- 230m 떨어진 출발지
- 1.42km 떨어진 출발지


9. 기존 예약 데이터 처리

이미 DB에 저장된 예약은 좌표 컬럼이 비어 있을 수 있습니다.

선택지는 두 가지입니다.

1. 좌표가 없는 예약은 거리 정렬 결과에서 제외
2. 좌표가 없는 예약은 목록 맨 뒤로 표시

처음 구현은 제외하는 방식이 단순합니다.

조건 예:

WHERE departure_lat IS NOT NULL
  AND departure_lng IS NOT NULL

단, 기존 예약도 계속 보여줘야 한다면 좌표 없는 예약은 distance_meters를 null로 두고 맨 뒤로 보내면 됩니다.


최종 작업 흐름 요약

1. reservations 테이블에 출발지/목적지 좌표 컬럼 추가
2. 예약 생성 API에서 좌표를 받도록 수정
3. 예약 생성 화면에서 Tmap 검색 결과의 latitude/longitude를 서버로 전송
4. 홈 화면에서 선택한 출발지/목적지를 NearbyMateList로 전달
5. 기존 /api/reservations/all API에 lat/lng 쿼리 파라미터 지원 추가
6. 백엔드에서 Haversine 공식으로 거리 계산
7. distance_meters 기준으로 가까운 순서 정렬
8. 목록 화면에서 거리 값을 함께 표시


핵심 결론

Tmap API는 그대로 사용합니다.
Tmap API는 장소를 검색해서 좌표를 얻는 역할이고,
거리 기반 동승자 정렬은 우리 백엔드가 DB 예약 좌표를 기준으로 처리해야 합니다.

따라서 새 외부 API가 필요한 것이 아니라,
기존 예약 생성/조회 흐름에 좌표 저장과 거리 계산 정렬을 추가하면 됩니다.

  중요: DB에는 실제 컬럼 추가가 필요합니다. 백엔드 .env의 DATABASE_URL이 맞게 설정된 상태에서 실
  행하세요.
